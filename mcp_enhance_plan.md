Step 1 — shared data helper. Both current tools independently call get_cash_rate(), compute China signal from the same three yfinance tickers, and fetch VIX. Extract this into a private _fetch_macro_core() function in a new data/macro_core.py or inline in the tools package. All subsequent tools call this once and receive a plain dict or small dataclass. Eliminates the duplicate HTTP calls.
Steps 2 & 3 — new models first, before touching tools. Define MacroRegime (enums and composite signals for the scorer) and MarketSnapshot (prices, series, news for display) as new Pydantic models in models/. Getting the schema right before touching the implementations means you can validate the shape of data independently. The old MacroInfo and MacroAnchors models stay in place until all callers are migrated — no flag day.
Step 4 — new tool implementations. Write get_market_snapshot() in tools/market_snapshot.py and get_macro_regime() in tools/macro_regime.py, both calling the shared helper from Step 1. The sector trend series currently embedded in MacroAnchors.sector_rotation stays in MacroRegime since it's contextual chart data that supports the regime narrative, not a display dashboard element.
Step 5 — simplify score_stock(). Its current signature is (technical, macro, anchors). After the refactor it becomes (technical, regime). The commodity availability check (if sum(...) >= 2: score += 20) gets removed entirely or replaced with a real commodity trend signal — e.g. copper 1-month momentum vs zero.
Step 6 — update analyze_stock(). Currently it calls both get_macro_info() and get_macro_anchors() and passes both downstream. After this step it calls only get_macro_regime() for scoring. If the frontend needs display data alongside analysis, that's fetched separately by the caller — not bundled into the analysis object.
Step 7 — server registration. Add the two new MCP tools to server.py, remove or alias the old ones. Update the docstrings carefully since Claude (and other MCP consumers) use those descriptions to decide which tool to call.
Step 8 — test the seam. The key thing to verify is that score_stock() still produces the same scores for the same market conditions, and that no network call happens more than once per analyze_stock() invocation. A simple script calling each tool and printing its output is enough at this stage.
